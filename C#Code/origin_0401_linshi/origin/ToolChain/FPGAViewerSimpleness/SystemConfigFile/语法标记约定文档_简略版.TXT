1、总则
1.1 所有的标记都以 //@ 开始。
1.2 所有的标记及其后的内容，必须在一行里完成。
1.3 所有的标记都是小写，其后的内容不分大小写。
1.4 所有的标记中的":"，为英文的":"字母，不要使用中文的"："。
1.5 所有的标记之后，须紧跟一个空格，然后再填写内容。
1.6 本方案总共分为4个方面的文档标记：工程、模块、模块IO、API寄存器。
其中，除API寄存器与软件有关外，其他供FPGA工程管理使用。
API寄存器涉及到自动生成软件代码工程中需要的寄存器定义文件，必须严格编写。

2、  有关工程的标记[//@project:]
2.1  所有的工程的标记，都以 //@project: 开头
2.2  工程的文档内容，应该只在一个.v文件中编写，往往是总控文件，也就是top文件。一般在top文件的头部进行编写和维护。  
2.3  //@project:partial_postfix_name 
      ★
      分FPGA 工程，生成的文档和软件接口文件的名称的附加部分。一个大的项目，可能由几个FPGA工程构成（如PCIE、处理板、采集板等），生成的文档和接口代码文件名称的后面部分进行附加，以便于进行区分。
      如PCIE工程，//@project:partial_postfix_name Pcie  ,则生成的文档文件名称可能是 XXXX_Pcie.html,生成的代码文件名称的后部可能是 XXXX_Pcie.cs
         处理板工程 ：//@project:partial_postfix_name ProcessBoard  ,则生成的文档文件名称可能是 XXXX_ProcessBoard.html ,,生成的代码文件名称的后部可能是 XXXX_ProcessBoard.cs

5、FPGA对外接口寄存器[//@apireg:]
5.1  该部分表示的是以地址定义的，与软件通过寄存器方式进行交互的文档。
5.2  该部分，一般在名称大致是XXXX_interface.v文件中。
5.3  ☆对该工具的价值，本功能才是其直接价值的体现。其他文档都是有关FPGA内部自己的事情，只与编写FPGA逻辑的人员相关，可能与软件编程人员不进行沟通交互。
     但需要与软件编程人员进行准确、及时、有效的沟通、约定的地方在此处。
     首先，没有人愿意编写完整、完备、准确的文档，这是人之常情。
     其次，在修改后，文档与FPGA的逻辑的对应，也是一件比较痛苦的事情。
     再次，文档文档的管理是一件繁杂的事情，往往不知所在，不知哪个是最新的版本。
     再次，经常出现软件端与FPGA端脱节的情况，也就是软件中存在FPGA中没有再使用的寄存器，而FPGA中新的寄存器
     最后，软件人员与软件是否按文档来进行编写，也是遥不可及的管理。
     也就是说，由于约定传输的路径长，人员不固定，没有固化，或者固化是非程序化的，都给FPGA逻辑编写人员和软件编写人员带来精力损耗，效率低下，还不可维护！
     本工具的目标是将原有的效率提高至少50倍以上，准确性提高90%以上。
     
     本工具的设计思路是，软件中有关寄存器的定义，以及必要说明，包括软件中对该寄存器的命名都是从FPGA代码文件中自动生成的。
     在软件系统中有关与FPGA的寄存器定义，约定，都100%依赖FGPA中的文档，软件中有关FPGA与软件交互的寄存器定义的代码文件，不能手动维护，只能自动生成！！！
     这意味着，FPGA工程中的文档编写必须是准确的！！！否则后续软件将导致错误。其实这也是符合常理的。
     
     这部分实际上定义了FPGA与软件的交互接口，如果处理不清楚接口交互，一切都是瞎忙活。
     为了方便管理，可以对寄存器进行分组管理。一个分组就是一个大的模块，比如控制模拟通道的，控制采集的，控制DDR的，控制触发的等等。在生成的文档中，可以清楚看出
     该模块有关的接口，从而大致检查接口的完备性和冗余性。

============有关接口分组的标识===========
5.5  有关接口分组的标识
     一个分组的定义，只需定义一次，要将某个接口定义归集到某个分组，只需要指明属于哪个分组就行了。
5.5.1 //@apireg:group:begin     
      寄存器分组开始标志。指明一个新的分组开始。用于划分为不同的分组。
5.5.2 //@apireg:group:title     
      分组的称呼，可是是汉字名称。仅仅用于精确的交流指定。
5.5.3 //@apireg:group:software:name  
      ★
      分组的软件名称。该名称是软件中寄存器命名的开头部分。同一个分组的寄存器，拥有同样的名称前缀，此处定义的就是这个名称前缀。
      生成软件定义文件时，自动将该名称作为软件寄存器名称的前缀。所以，应该短小、精炼、明确，并符合软件取名规则[不能有汉字、空格等其他不可用字符]。
      除首次定义处外，如果在FPGA代码中不是连续的组，在其它地方归集时，以此标志为指定归集的组。
5.5.4 //@apireg:group:end   
      ★
      一个组的结束标志。在一个新的组开始前标志。可以没有此标志。是为了文档清晰。
5.5.5 //@apireg:group:desc   
      该组的详细描述。
================================

5.6   //@apireg:write_read_attribute:attribute    
      ★
     指示寄存器是读寄存器还是写寄存器，一般在首个翻译读写寄存器的前面编写一句即可。后续的继承此属性，直到修改。
     //@apireg:write_read_attribute:attribute read
     或
     //@apireg:write_read_attribute:attribute write

5.7  有关软件接口的标识。
     本处的接口，不是FPGA中模块的IO接口，而是类似于XXXX_interface.v文件中，case (xxx) \r\n 8'h04:这样的寄存器地址的定义。
     这些接口负责与软件的交互。往往涉及两个重要的方面：地址和值的约定。
     接口的文档是纯文档的，目前与代码没有“硬”关联。也就是说，可以将文档写在任何一个地方。但是，我们强烈要求，写在该寄存器解析的地方，也就是软件地址约定的地方，一般写在其前一行。
5.7.1 //@apireg:title 
      ★
     寄存器称呼。用于双方交流明确指向。可以是中文名称。
5.7.2 //@apireg:desc 
     寄存器的描述。往往什么寄存器的用途，必要时可以说明后续涉及的模块。                               
5.7.3 @apireg:note 
     注意事项。有重要的注意事项时，需要说明。FPGA人员不要以为软件人员什么都清楚，也不要以为FGPA人员什么都能记住。所以，重要的事项一定要记录下来。
5.7.4 //@apireg:0xaddr 
      ★
     寄存器的地址。该地址是以16进制编写的。由于地址往往经过不同的解码，可能软件使用的地址与FPGA中的值不是绝对一致。软件人员并不知道转换方法，所以FPGA人员应该编写软件端使用的地址。
5.7.5 //@apireg:software:name 
      ★
     寄存器在软件代码中使用的名称。该名称与该寄存器所属的组名称一起构成软件名称。
     比如一个寄存器的定义为 //@apireg:software:name ReadEnable   ，其组为////@apireg:group:software:name DDR，则自动生成的软件名称为 DDR_ReadEnable。
     一般情况下，使用大驼峰规则来命名。
5.7.6 //@apireg:software:note 
     该寄存器软件需要注意的内容。
5.7.7 //@apireg:value:appoint 
      ★
     该寄存器取值的约定。一般情况下，寄存器的值分3种情形：
     一种是只有0和1取值的：此时需要说明是状态寄存器还是用于产生沿的寄存器。如果是沿寄存器，需要说明什么沿有效。
     一种是多位组合的：此时需要指明每一位的含义和高低约定。
     一种是代表数字的：此时需要指明取值范围（可以按有效位数来说明），还需要指明该数字代表的含义。比如是312.5M（3.2ns）时钟个数什么的。一定要有单位。
5.7.8 //@apireg:notused   标记该寄存器目前暂时没有使用
     此类寄存器，将不在软件中出现。标记为此类的寄存器，需要标记地址，其他可以不标记。
     如：
     //@apireg:notused 
     //@apireg:0xaddr 0x8800|(0x02c<<2)	
5.7.9 //@apireg:total:image  标记对软件寄存器的理解有帮助用的原理示意图。一般放在寄存器解码前的always 语句前     
示例：
      //@apireg:write_read_attribute:attribute write
			case(adsp_addr_decode[7:0])

===============      有关软件接口最佳实践建议       =====================
=========================================================================
下面是软件寄存器接口的最佳实践的有关建议：
1、尽可能在一个.v文件中定义与软件接口的寄存器（虽然这不是必须的）。
    这主要是避免两个问题：
   第一：定义在不同的.v文件中，不好查找（对FPGA编程人员也是一样的）；
   第二：避免地址重复，以及不同的编码方式的不一致。
2、最好将相同模块或实现完整功能的寄存器组织在一块。（虽然这不是必须的）。
   这可以带来两个好处：
   第一：可以清楚看到该模块或功能有关的操作是否完备，是否缺乏或多余。如果穿插在不同的地方，从FPGA编程来讲，一个模块或功能的实现有关的接口显得凌乱，没有整体性。
   第二：可以减少//@apireg:group:title 和//@apireg:group:end标签的使用，从而减少文件的代码行数。
3、不同同模块或实现类似功能的寄存器的地址分配，最好留一定的地址间隔（5个以上）。
  这样的好处，是为该模块将来寄存器的扩展留一定的空间，而不修改或调整其他已经存在的寄存器的地址。
4、在必要的时候，可以对寄存器的地址进行整体规划和修改，以便逻辑的完整性和清晰性。
   使用现有的方式，软件端没有绝对的地址，不影响代码的逻辑和编译。因此，寄存器的地址仅仅是一个符号，可以随意修改而不影响软件。
5、最好将寄存器按模块或功能实现进行分组，除测试用的寄存器外，不能分组的寄存器应该很少。
   分组的目的是为了对系统的功能或模块有一个全貌的展示，而不是凌乱。
6、一个寄存器最少的标签应该包括 软件名称和地址。这是本方式使用的额必要条件。
    除连续数外（如一个48的数，使用Low16,M16,H16三个寄存器来传输），每个寄存器还应该标识tilte和appoint。
7、//@apireg:desc 和 //@apireg:note 标签一般不编写。
8、//@apireg:value:appoint 往往是必须的。
    一般//@apireg:value:appoint的说明的内容包括：
   1）需要表明该寄存器的值是位控型的还是非位控型的。
   2）对只有1位的寄存器，要表明是状态型的还是边沿有效型。边沿有效型的，要说明是什么边沿有效，该边沿用来干什么。若有数据准备的，要说明先后顺序。这种寄存器要说明等待延时的考虑依据。
   3）对多位位控的，要说明是独热码型的（只有一位有效，其取值而2^N）还是各位控制的。
   4）对独热码，要列出每个码的含义。
   5）对各位控制的，要列出每一位控制什么。
   6）对非位控型的，又分为枚举类型的和任意取值的。对枚举类型的，其每一个值在FPGA中都有特殊的处理代码或流程，应该分别列出其枚举值可选列表和含义。
9、对需要特殊控制延时的，有两种处理方式，一种是判断处理完毕的方式，一种是固定延时的方式。
   使用寄存器读取回来判断处理完毕的方式，适用于需要长时间延时等待或者等待时间不确定的情况。
   一般情况下，在软件端延时固定时间的方式。此时，需要明确说明延时的时间长短（以uS为单位），如果是SPI传输，可以通过SPI的时钟和传输的数据字节数计算出来。
   等待SPI传输完毕的，可以说明SPI的时钟和传输字节数，软件端好计算延时时间。
10、流程与场景描述。
11、最好将本工程中，需要软件控制的模块的定义，集中编写在一起。
      这样的好处是，可以一目了然地看清楚本工程涉及到的模块。如果在其他地方定义，则杂乱无章和不好引用，也不好明确各个模块的软件前缀。
      同时，可以减少出错。在寄存器模块归类时，只需要引用 //@apireg:group:title XXXX就可以了。
      一般放在软件接口寄存器地址解码前的第一个always之前。这样可以顶行编写，免于出现格式的问题。
12、一个PFGA工程，最好有一个该工程的原理框图，重点说明与软件有关的模块，这样都比较清楚软件需要控制的模块，也可以帮助FPGA工程人员清晰整理接口。
13、在接口文件的有关寄存器读写部分，尽可能不在编写附加的注释。
      第一，有关分块的注释，可以使用//@apireg:group:title 和 //@apireg:group:end 来标记。
      第二，有关寄存器值与用法等方面的注释，可以放在//@apireg:desc，//@apireg:value:appoint，//@apireg:note，//@apireg:software:note 等标签中。
      第三，放在标签中的信息，可以共享和固化。并可以自动通过使用人员的使用来验证其正确性、准确性和有效性。
14、有关软件接口使用的最多的标签包括：
      @project:partial_postfix_name   说明：现有的系统比较复杂，一个系统往往存在多个FGPA工程，为了区分不同的FPGA工程的寄存器，要进行区分。其结果往往是：第一，生成包含有此后缀的文件名称；第二，类名往往以也包含此后缀。一般放在Interface .v文件的头部。
      @apireg:total:image  说明：往往标识一个原理图。往往也是必须的，否则对该FPGA系统云里雾里。一般放在寄存器解码前的always 语句前。其路径以FPGA工程文件所在的目录（Ise为 XXX.xise,Vivado 为XXX.xpr）为相对路径。一般放在doc目录下，路径为 .\doc\xxx.png。

      //建议在Interface 文件的地址解码的代码的always语句 前集中定义各个模块。在寄存器地址解码处引用模块title。
      //@apireg:group:begin  
      //@apireg:group:title
      //@apireg:group:software:name
      //@apireg:group:end
   
      //有关寄存器定义用的标签
      //@apireg:title 寄存器名称，对该寄存器的称呼约定
      //@apireg:desc  该标签不是必须的，对复杂逻辑或功能的，需要说明。
      //@apireg:0xaddr 必须的。按每个FGPA工程地址译码的规则编写
      //@apireg:software:name  必须的。第一次命名时，需要与软件人员一起确定。如果到了工程后期，就不要修改了。软件中的名称包含了模块或分组的名称作为前缀。
      //@apireg:value:appoint 基本上是必须的。
    
      //读写类寄存器标识。这里的读写是从软件的角度看的。
      //@apireg:write_read_attribute:attribute write  一般放在对写入类的首个寄存器前面，只要一句。
      //@apireg:write_read_attribute:attribute read  一般放在对读取类的首个寄存器的前面，只要一句。
